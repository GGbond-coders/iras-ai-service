## 1. 项目概述

### 1.1 项目简介

`iras-ai-service` 是 IRAS 智能简历诊断系统的 **AI 服务层**，使用 Python 语言构建，替代原项目中的 Dify AI 平台。提供与 Dify 100% 兼容的 API 接口，原 IRAS 系统只需修改一行配置即可完成切换。

### 1.2 核心功能

| 功能 | 说明 |
|------|------|
| **职能画像** | 输入职位名称，LLM 结合 ChromaDB 向量库中相似岗位 JD，生成结构化能力画像（硬技能、软技能、工具、经验、学历） |
| **简历诊断** | 上传简历文件，LLM 语义检索匹配岗位并逐一评估，生成包含匹配分、差距分析、面试建议的诊断报告 |
| **知识库管理** | 基于 ChromaDB 的向量存储，岗位 JD 自动向量化索引，支持**增量同步**（基于时间戳）和**孤儿向量清理** |
| **岗位数据管理** | 支持种子数据初始化、CSV 批量导入、管理后台 CRUD 操作，数据变更通过增量同步更新向量库 |
| **Dify 兼容** | API 接口完全兼容 Dify 格式（SSE 流式响应），原 IRAS 系统无需代码改动即可切换 |

### 1.3 与原项目的差异

| 对比维度 | 原 Dify 方案 | 本服务方案 |
|----------|-------------|-----------|
| AI 平台 | Dify 云服务 | 自建 Python FastAPI |
| 知识库更新 | 手动上传文档 | 自动增量同步（MySQL → ChromaDB） |
| 部署依赖 | 外部 API 依赖 | 仅需通义千问 API Key |
| 数据控制 | 数据存储在 Dify 云端 | 数据完全本地化 |
| 扩展性 | 受限于 Dify 平台能力 | 完全自主可控 |

### 2.2 数据流

**职能画像流程**：
```
用户输入职位名 → /v1/workflows/run → ChromaDB 检索相似 JD → LLM 生成画像 → SSE 流式返回
```

**简历诊断流程**：
```
用户上传简历 → /v1/files/upload → 保存并返回 file_id
用户发起诊断 → /v1/workflows/run → 解析简历文本 → LLM 提取检索关键词 → ChromaDB 语义检索匹配岗位 → LLM 逐一评估 → SSE 流式返回
```

**岗位数据同步流程**：
```
管理后台 CRUD → MySQL 写入（含 create_time/update_time/status）
管理后台点击"同步向量库" → Spring Boot 代理 → Python /admin/chromadb/sync
  → 增量 upsert 有变化的岗位（≤10条/批）
  → 清理 MySQL 中已删除的孤儿向量
  → 返回统计信息（更新 N 条 + 清理 M 条残留）
```

## 3. 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| Web 框架 | FastAPI | HTTP API + SSE 流式响应 |
| ASGI 服务器 | Uvicorn | 生产级异步服务 |
| LLM 编排 | LangChain | Prompt 模板、流式调用 |
| 大语言模型 | 通义千问 qwen-plus | 职能画像生成、简历诊断分析 |
| 嵌入模型 | 通义千问 text-embedding-v3 | 文本向量化（每批最多 10 条） |
| 向量数据库 | ChromaDB | 岗位 JD 语义检索（余弦相似度） |
| ORM | SQLAlchemy | MySQL 数据库操作 |
| 文档解析 | PyPDF2 + python-docx | 简历文件解析（PDF/DOCX/TXT） |
| CSV 处理 | csv 标准库 | 批量导入岗位数据 |
| 数据库驱动 | PyMySQL | MySQL 连接 |

> **注意**：v1.0 中的数据采集器（httpx + BeautifulSoup4 + APScheduler）已移除。原因是国内主流招聘网站（51job、BOSS直聘、智联招聘、24365 等）均设有严格的反爬机制（SSO 登录墙、验证码、IP 频率限制），自动化采集不可行。岗位数据改为通过 CSV 批量导入 + 管理后台手动添加的方式维护。

## 5. API 接口设计

本服务对外暴露的 API 完全兼容 Dify Workflow API 格式。

### 5.1 认证

所有 `/v1/` 前缀接口需要 `Authorization: Bearer <token>` 请求头，token 在 `config.py` 的 `SERVICE_TOKEN` 中配置。管理接口（`/admin/` 和 `/health`）无需认证。

### 5.2 POST /v1/workflows/run

执行 AI 工作流（职能画像或简历诊断）。

**请求体**：
```json
{
  "inputs": {
    "job_name": "Java开发工程师"
  },
  "response_mode": "streaming",
  "user": "iras-user"
}
```

**inputs 字段路由规则**：
- 含 `job_name` → 职能画像工作流
- 含 `resume_text` → 简历诊断工作流（值为文件引用数组 `[{"upload_file_id":"xxx",...}]`）
- 都不含 → 400 错误

**响应**：SSE 流式输出（`text/event-stream`），事件类型：

| 事件 | data 格式 | 说明 |
|------|-----------|------|
| `workflow_started` | `{"id":"uuid"}` | 工作流开始 |
| `text_chunk` | `{"text":"..."}` | 文本片段，逐 chunk 输出（前端打字机效果） |
| `workflow_finished` | `{"outputs":{"result":"..."}}` | 工作流完成，result 为最终 JSON |
| `[DONE]` | — | 流结束标记 |

SSE 格式严格要求：`data: {"event":"xxx","data":{...}}\n\n`，每个事件以双换行分隔，JSON 内中文不转义（`ensure_ascii=False`）。

### 5.3 POST /v1/files/upload

上传简历文件。

**请求**：multipart/form-data
- `file`：文件二进制（PDF/DOCX/TXT）
- `user`：用户标识（默认 `iras-user`）

**响应**：
```json
{
  "id": "uuid-xxxxx"
}
```

文件存储在 `./uploads/` 目录，文件名为 `{uuid}{原始扩展名}`。file_id 用于后续 workflow 调用中的 `resume_text` 参数。

### 5.4 GET /health

健康检查端点，供外部监控探活。

```json
{"status":"ok"}
```

### 5.5 POST /admin/chromadb/sync

增量同步 ChromaDB 向量库。从 MySQL 读取上次同步后有变化的岗位，upsert 到 ChromaDB，同时清理 MySQL 中已不存在的孤儿向量。

**响应（成功）**：
```json
{"status":"ok","message":"更新 150 条，清理 2 条残留"}
```

**响应（失败）**：
```json
{"status":"error","message":"具体错误信息"}
```

此端点由 Spring Boot 管理后台的 `AdminController.syncChromaDb()` 代理调用，前端管理面板通过"同步向量库"按钮触发。

## 6. AI 工作流设计

### 6.1 LLM 配置

使用通义千问的 OpenAI 兼容接口：

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(
    model="qwen-plus",                                               # 性价比最优
    api_key="sk-xxx",
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    temperature=0.3,                                                  # 低温度确保 JSON 格式稳定
)
```

### 6.2 职能画像工作流

**输入**：职位名称（如 "Java开发工程师"）

**流程**：
1. 将职位名称作为查询，在 ChromaDB 中检索 top-5 相似岗位 JD
2. 构建 Prompt：包含职位名称 + 参考 JD 上下文（可为空）
3. LLM 流式生成 5 个维度的结构化 JSON

**输出格式**：
```json
{
  "job_title": "Java开发工程师",
  "hard_skills": ["Java", "Spring Boot", "MySQL", "Redis", "微服务"],
  "soft_skills": ["团队协作", "问题分析", "沟通表达"],
  "tools": ["Git", "Maven", "Docker", "Jenkins", "IntelliJ IDEA"],
  "experience": "3-5年Java开发经验，有大型项目经验优先",
  "education": "本科及以上，计算机相关专业"
}
```

### 6.3 简历诊断工作流

**输入**：简历纯文本 + ChromaDB 岗位库

**流程**：
1. 解析上传的简历文件（PDF/DOCX/TXT → 纯文本）
2. **LLM 从简历中提取检索关键词**（过滤噪音，保留技能/职位/行业/资历维度，输出 20-50 字检索串）
3. 用提取的**关键词**在 ChromaDB 语义检索 top-10 匹配岗位（替代旧版简历全文检索，精度更高）
4. 构建 Prompt：包含简历全文 + 岗位列表及 JD（每个 JD 截取前 600 字符）
5. LLM 流式生成诊断报告（匹配 1-3 个岗位逐一评估）

**输出格式**：
```json
[
  {
    "matched_job": "Java后台开发工程师（上海联泉智能科技有限公司）",
    "matched_score": 85,
    "matched_reason": "候选人的Java和Spring Boot经验与岗位高度契合",
    "gap_points": ["缺少Spring Cloud微服务经验", "未体现性能调优能力"],
    "interview_advice": ["准备微服务架构相关问答", "复习JVM调优知识"],
    "type": "全职",
    "company": "上海联泉智能科技有限公司",
    "city": "上海",
    "salary": "15000"
  }
]
```

诊断结果使用 `<think>...</think>` 标签分离推理过程和结构化输出，前端可选择展示/隐藏推理过程。

### 6.4 简历文件解析

支持格式及解析方式：

| 格式 | 解析库 | 方法 |
|------|--------|------|
| `.txt` / `.text` | 内置 open() | UTF-8 直接读取，解码失败回退 GBK |
| `.pdf` | PyPDF2.PdfReader | 逐页 `extract_text()` 后拼接 |
| `.docx` | python-docx.Document | 遍历 `paragraphs` 提取 text |

### 6.5 SSE 流式输出的"收集再分发"模式

`_run_job_profile()` 和 `_run_resume_diagnosis()` 均采用"先收集全部 LLM 输出片段，再分发 SSE 事件"的策略：

```
LLM 流式输出 → 收集所有片段到列表 → 除最后一片外作为 text_chunk 发出（打字机效果）
                                   → 最后一片（完整清洗后的 JSON）作为 workflow_finished 发出
```

**设计原因**：Dify 的 SSE 协议要求 `workflow_finished` 事件携带完整的最终结果。如果每收到一个 token 就发 `text_chunk`，最后一片（完整 JSON）也会被当作 `text_chunk` 发出，导致 `workflow_finished` 无内容可发。

#### diagnosis_record 表

| 字段 | 类型 | 说明 |
|------|------|------|

## 8. 知识库设计

### 8.1 向量库架构

- **引擎**：ChromaDB（本地持久化，`./chroma_data/` 目录）
- **嵌入模型**：通义千问 text-embedding-v3（远程 API，DashScope 限制每次最多 10 条文本）
- **相似度算法**：余弦相似度（cosine），适合长度差异大的中文文本匹配
- **Collection 名称**：`iras_jobs`
- **v1 策略**：一条 JD 对应一个向量文档，不做 chunk 分块

### 8.2 嵌入函数选择

使用 ChromaDB 的 `OpenAIEmbeddingFunction` + 阿里云 DashScope，而非 ChromaDB 默认的本地模型（all-MiniLM-L6-v2）：

- **避免启动时下载 80MB 模型文件**，缩短冷启动时间
- **中文嵌入质量更优**，text-embedding-v3 在中文语义理解上显著优于轻量本地模型
- **与 LLM 同源**，嵌入和生成使用同一厂商 API，减少语义空间不一致

代价：每次写入向量库时需调用远程嵌入 API（有网络延迟和费用），但读取（查询）时使用已存储的向量，不影响检索速度。

### 8.3 向量文档结构

```python
{
    "id": str(job.id),                    # 与 MySQL job_info.id 一一对应
    "document": job.jd_text,               # JD 全文
    "metadata": {
        "job_name": job.job_name,
        "company_name": job.company_name,
        "city": job.city,
        "salary": job.salary,
    }
}
```

### 8.4 增量同步策略

核心函数 `sync_jobs()` 替代了 v1.0 的全量重建（`rebuild_all()`），实现高效的增量同步：

```
读取上次同步时间戳（last_sync.txt）
  ├── 时间戳为 0（首次同步）
  │     → upsert 全部 status='active' 的岗位
  │
  └── 时间戳 > 0（增量同步）
        ├── upsert 自上次以来有变化的 active 岗位（create_time/update_time >= last_sync）
        ├── 删除状态变为非 active 的岗位向量（软删除）
        └── 清理孤儿向量（ChromaDB 中有但 MySQL active 中不存在的 ID，即硬删除）
            → 对比 ChromaDB 现有 ID 集合与 MySQL active ID 集合，清理差集
```

**批次控制**：`BATCH_SIZE = 10`，将岗位分 10 条一批调用嵌入 API，避免超出 DashScope 限制。upsert 操作是幂等的（ID 存在则更新，不存在则新增）。

**同步时间戳文件**：`{CHROMA_PERSIST_DIR}/last_sync.txt`，记录上次同步完成时的 Unix 时间戳。

### 8.5 操作接口

| 操作 | 函数 | 触发场景 |
|------|------|----------|
| 增量同步 | `sync_jobs()` | 管理后台点击"同步向量库" |
| 批量添加 | `add_jobs(ids)` | （预留）单岗位即时同步 |

### 13.2 路由层 — routes/workflows.py

**文件**：`routes/workflows.py`

这是整个服务最核心的路由文件，实现了与 Dify 100% 兼容的 SSE 流式响应。

#### 13.2.1 SSE 编码函数

```python
def sse_encode(event) -> str:
    data = json.dumps(event.__dict__, ensure_ascii=False)
    return f"data: {data}\n\n"
```

**格式要求**（必须精确匹配 Dify）：
- `data: ` 前缀 + JSON + `\n\n`（双换行）
- `ensure_ascii=False`：中文不转义为 `\uXXXX`，前端直接可读
- 每个事件是一个普通类实例，通过 `.__dict__` 序列化（普通类的 `__dict__` 更干净，无 Pydantic 私有属性干扰）

#### 13.2.2 Token 验证

```python
async def verify_token(request: Request) -> None:
    authorization = request.headers.get("Authorization", "")
    if not authorization.startswith("Bearer "):
        raise HTTPException(status_code=401, detail="Missing or invalid token")
    if authorization.removeprefix("Bearer ") != SERVICE_TOKEN:
        raise HTTPException(status_code=401, detail="Invalid token")
```

`SERVICE_TOKEN` 在 `config.py` 中定义（默认 `iras-ai-service-token-2024`），与 Spring Boot 后端 `application.yml` 中的 `dify.job-profile-key` 和 `dify.resume-diagnosis-key` 一致。两端共享同一个 token 完成认证。

#### 13.2.3 工作流路由分发

```python
@router.post("/workflows/run")
async def run_workflow(request: WorkflowRequest, req: Request):
    await verify_token(req)
    inputs = request.inputs

    if "job_name" in inputs:
        generator = _run_job_profile(inputs["job_name"])
    elif "resume_text" in inputs:
        file_ref = inputs["resume_text"]
        file_id = None
        if isinstance(file_ref, list) and len(file_ref) > 0:
            file_id = file_ref[0].get("upload_file_id", "")
        generator = _run_resume_diagnosis(file_id)
    else:
        raise HTTPException(status_code=400, detail="Unknown workflow inputs")
```

**路由逻辑**：
- `job_name` 存在 → 职能画像工作流
- `resume_text` 存在 → 简历诊断工作流（从嵌套的 `upload_file_id` 中提取文件 ID）
- 都不存在 → 400 错误

与 Dify inputs 格式完全兼容：Dify 的 inputs 是自由 JSON 对象，key 由 workflow 定义，本服务据此判断调用方意图。

#### 13.2.4 SSE 事件流包装

```python
async def event_stream():
    async for chunk in generator:
        yield chunk
    yield "data: [DONE]\n\n"

return StreamingResponse(
    event_stream(),
    media_type="text/event-stream",
    headers={
        "Cache-Control": "no-cache",
        "Connection": "keep-alive",
        "X-Accel-Buffering": "no",   # 禁用 Nginx 反向代理缓冲
    },
)
```

- `StreamingResponse` + `text/event-stream`：标准 SSE 响应类型
- `X-Accel-Buffering: no`：如果前端有 Nginx 反代，禁用缓冲确保流式输出不被截断
- `[DONE]` 信号：流结束标记，Spring Boot 的 `DifyServiceImpl.parseSseStream()` 据此终止读取

#### 13.2.5 职能画像生成器（_run_job_profile）

```python
async def _run_job_profile(job_name: str):
    workflow_id = str(uuid.uuid4())
    yield sse_encode(WorkflowStartedEvent(workflow_id))   # 1. 开始事件

    try:
        all_pieces = []
        async for piece in generate_job_profile(job_name, vector_store=vector_store):
            all_pieces.append(piece)                       # 2. 收集所有流式片段

        for piece in all_pieces[:-1]:
            yield sse_encode(TextChunkEvent(piece))        # 3. 除最后一片外→text_chunk
            await asyncio.sleep(0.01)

        final = all_pieces[-1] if all_pieces else ""
        yield sse_encode(WorkflowFinishedEvent(final))     # 4. 最后一片→workflow_finished
    except Exception as e:
        yield sse_encode(WorkflowFinishedEvent(            # 5. 异常→workflow_finished(error)
            json.dumps({"error": str(e)}, ensure_ascii=False)))
```

**"收集再分发"模式的设计原因**：

`generate_job_profile()` 通过 `yield` 逐 token 输出 LLM 流式结果。但 Dify 的 SSE 协议要求 `workflow_finished` 事件携带**完整的最终结果**。如果每收到一个 token 就发 `text_chunk`，最后一片（完整 JSON）也会被当作 `text_chunk` 发出，导致 `workflow_finished` 无内容可发。

因此采用"先收集全部，最后一片作为 final，其余作为 text_chunk"的策略：
- `text_chunk` 事件：让前端展示"打字机"流式效果（仅限中间推理过程）
- `workflow_finished` 事件：携带最终结构化 JSON 供前端解析

`asyncio.sleep(0.01)` 的作用：让出事件循环控制权，确保 SSE 块能被及时 flush 到客户端，避免浏览器缓冲导致卡顿。

#### 13.2.6 简历诊断生成器（_run_resume_diagnosis）

流程与职能画像类似，但增加了简历解析步骤：

```
1. yield workflow_started
2. 检查 file_id → 为空则 yield workflow_finished(error)
3. parse_resume(file_id) → 失败则 yield workflow_finished(error)
4. diagnose_resume(resume_text, vector_store) → 收集片段
5. yield text_chunk（除最后一片）
6. yield workflow_finished（最后一片 = wrapped JSON）
```

**错误处理分级**：
- 文件 ID 缺失 → 返回空诊断结果（含错误信息的 JSON 数组）
- 文件解析失败 → 返回含错误信息的诊断结果
- LLM 调用失败 → 返回 `{"error": "..."}`

所有异常都包装为 `workflow_finished` 事件而非 HTTP 错误，因为 SSE 流一旦开始，HTTP 状态码已发送（200），只能在流内传递错误。

#### 13.4.1 Prompt 设计

```python
JOB_PROFILE_PROMPT = ChatPromptTemplate.from_template("""你是一位资深的HR专家...
职位名称：{job_name}
{job_context}
请严格按照以下JSON格式输出...
{
  "job_title": "{job_name}",

**Prompt 设计考量**：
- `{job_context}` 占位符：可选插入 ChromaDB 检索到的相关 JD 作为参考。为空时 LLM 基于自身知识生成，仍能产出合理的画像。

- `job_title` 保持与输入一致：防止 LLM 自行修改职位名称。
- 明确的数组长度要求（4-8、3-5、5-8）：防止 LLM 输出过于简略或过度膨胀。

#### 13.4.2 生成函数

```python
async def generate_job_profile(job_name: str, vector_store=None):
    # 1. 构建 job_context（可选的向量检索上下文）
    job_context = ""
    if vector_store:
        results = vector_store.search(job_name, top_k=5)
        if results:
            job_context = "以下是与该职位相关的参考岗位描述：\n" + ...

    # 2. LLM 流式生成
    llm = _build_llm()
    prompt = JOB_PROFILE_PROMPT.format(job_name=job_name, job_context=job_context)
    full_text = ""
    async for chunk in llm.astream(prompt):
        text = chunk.content
        full_text += text
        yield text                    # 逐 token yield 给上层

    # 3. 清洗输出（去掉可能的 markdown 代码块标记）
    clean = full_text.strip()
    if clean.startswith("```"):
        lines = [l for l in clean.split("\n") if not l.startswith("```")]
        clean = "\n".join(lines).strip()
    yield clean                        # 最后 yield 完整清洗结果
```

**两层 yield 的含义**：
- `yield text`（循环内）：LLM 原始流式 token，由 `_run_job_profile` 收集为 `text_chunk` 事件发给前端
- `yield clean`（最后）：完整清洗后的 JSON 字符串，由 `_run_job_profile` 作为 `workflow_finished` 事件的 result 发出

**为什么需要最后的 `yield clean`**：LLM 流式输出的拼接结果可能包含 markdown 代码块标记（\`\`\`json ... \`\`\`），必须清洗后重新发出完整 JSON。

```python
# v1.2（旧）: 简历全文直接检索
results = vector_store.search(resume_text, top_k=10)

# v1.3（新）: 先提取关键词，再检索
search_query = await extract_keywords(resume_text)   # "Java Spring Boot 后端开发 3年 本科"
results = vector_store.search(search_query, top_k=10)
```

**检索精度提升原理**：
- 简历全文 ~ 1500 字，含大量与岗位匹配无关的信息（个人信息、项目描述）
- 关键词串 ~ 30 字，只包含技能/职位/行业/资历四个维度的核心信号
- ChromaDB 的语义向量对长文本会平均化各维度信息，短而精的关键词查询能聚焦核心语义
- `top_k=10`：给 LLM 提供足够的候选岗位，由其从中筛选 1-3 个最匹配的

#### 13.5.3 结果包装

```python
    wrapped = json.dumps({"result": clean}, ensure_ascii=False)
    yield wrapped
```

**`{"result": "..."}` 包装的原因**：前端 `Diagnosis.vue` 和 `History.vue` 的解析逻辑期望顶层 JSON 包含 `result` 字段。前端解析流程：

1. `JSON.parse()` → 得到 `{result: "<think>...</think>\n[{...}]"}`
2. 检测到 `parsed.result` 字段 → 进入解包分支
3. 正则提取 `<think>` 内容 → 存入 `thinkContent`
4. 去掉 `<think>` 标签后 `JSON.parse()` 内层数组 → 得到匹配岗位列表
5. 渲染岗位卡片

Spring Boot 后端 `DifyController.saveDiagnosisRecord()` 也依赖此格式：通过 `root.has("result")` 检测包装，提取内层 JSON 计算 `matchCount`。

```python
def parse_resume(file_id: str) -> str:
    # 在 uploads 目录查找匹配 file_id 前缀的文件
    for filename in os.listdir(upload_dir):
        if filename.startswith(file_id):
            filepath = os.path.join(upload_dir, filename)
            ext = os.path.splitext(filename)[1].lower()
            if ext in (".txt", ".text"):
                return _parse_txt(filepath)     # UTF-8 读取（非法字节 replace）
            elif ext == ".pdf":
                return _parse_pdf(filepath)     # PyPDF2 逐页提取
            elif ext in (".doc", ".docx"):
                return _parse_docx(filepath)     # python-docx 逐段落提取
            else:
                raise RuntimeError(f"Unsupported file type: {ext}")
    raise FileNotFoundError(f"Uploaded file not found: {file_id}")
```

**文件查找方式**：file_id 是上传时生成的 UUID，实际文件名为 `{uuid}{原始扩展名}`。通过前缀匹配查找确保即使扩展名变化也能定位。

**三种解析器**：

| 解析器 | 库 | 方法 | 容错 |
|--------|-----|------|------|
| TXT | 内置 open() | UTF-8 读取全文 | `errors="replace"` 处理非法字节 |
| PDF | PyPDF2.PdfReader | 逐页 `extract_text()` 后拼接 | 空页返回 "" |
| DOCX | python-docx.Document | 遍历 `paragraphs` 提取 text | 异常包装为 RuntimeError |

#### 13.7.3 增量同步核心 — sync_jobs()

```python
def sync_jobs():
    last_sync = _read_last_sync()

    if last_sync == 0.0:
        # 首次同步：全量 upsert 所有 active 岗位
        jobs = db.query(JobInfo).filter(JobInfo.status == "active").all()
    else:
        # 增量同步：仅处理自上次同步以来有变化的岗位
        last_dt = time.strftime("%Y-%m-%d %H:%M:%S", time.localtime(last_sync))
        jobs = db.query(JobInfo).filter(
            JobInfo.status == "active",
            or_(JobInfo.create_time >= last_dt, JobInfo.update_time >= last_dt),
        ).all()

        # 软删除处理
        expired = db.query(JobInfo).filter(
            JobInfo.status != "active",
            JobInfo.update_time >= last_dt,
        ).all()
        if expired:
            delete_jobs([j.id for j in expired])

    if jobs:
        _batch_upsert(jobs)  # 分批写入，每批 ≤10 条

    # 孤儿清理：对比 ChromaDB ID 与 MySQL active ID
    active_ids = {str(r[0]) for r in db.query(JobInfo.id).filter(JobInfo.status == "active").all()}
    existing = collection.get()
    orphans = [vid for vid in existing["ids"] if vid not in active_ids]
    if orphans:
        collection.delete(ids=orphans)

    _write_last_sync(time.time())
    return {"jobs_upserted": len(jobs), "orphans_cleaned": len(orphans)}
```

**关键设计点**：
- **时间戳增量**：首次同步（时间戳=0）全量写入，后续仅处理变化的岗位，大幅减少嵌入 API 调用量
- **批次控制**：`_batch_upsert()` 将岗位按 10 条一批分片调用嵌入 API，遵守 DashScope 限制
- **孤儿清理**：对比 ChromaDB 现有 ID 集合与 MySQL active ID 集合的差集，处理 Spring Boot 硬删除（DELETE）产生的孤儿向量
- **时间戳文件**：持久化在 `{CHROMA_PERSIST_DIR}/last_sync.txt`，保证服务重启不丢失同步状态

```python
def search(query: str, top_k: int = 5) -> list[dict]:

#### 13.8.1 database.py — 引擎与会话

DATABASE_URL = (

**连接池参数**：
- `pool_pre_ping=True`：每次从池中取出连接时先 ping 验证，避免使用已被 MySQL 端关闭的死连接
- `pool_recycle=3600`：每小时强制回收连接。MySQL 默认 `wait_timeout=28800`（8 小时），1 小时回收远早于超时
- `autoflush=False`：手动控制 flush 时机，避免读取操作触发意外的数据库写入

```python
def get_db() -> Session:

```python
class WorkflowRequest(BaseModel):
    inputs: dict[str, Any]           # Dify 兼容的 inputs 自由 JSON
    response_mode: str = "streaming"  # 固定 streaming
    user: str = "iras-user"

class FileUploadResponse(BaseModel):
    id: str = Field(default_factory=lambda: str(uuid.uuid4()))

class WorkflowStartedEvent:
    def __init__(self, workflow_id):
        self.event = "workflow_started"
        self.data = {"id": workflow_id}

class TextChunkEvent:
    def __init__(self, text):
        self.event = "text_chunk"
        self.data = {"text": text}

class WorkflowFinishedEvent:
    def __init__(self, result):
        self.event = "workflow_finished"
        self.data = {"outputs": {"result": result}}
```

**Event 类使用普通类而非 Pydantic BaseModel**：SSE 编码函数 `sse_encode()` 使用 `event.__dict__` 序列化。普通类的 `__dict__` 直接包含 `{"event": "...", "data": {...}}`，而 Pydantic BaseModel 的 `__dict__` 会包含额外的私有属性。使用普通类更轻量且输出格式更干净。

**SSE 事件的 JSON 结构**：
```
工作流开始:  {"event": "workflow_started",   "data": {"id": "uuid"}}
文本片段:    {"event": "text_chunk",         "data": {"text": "..."}}
工作流完成:  {"event": "workflow_finished",  "data": {"outputs": {"result": "..."}}}
```

注意 `workflow_finished` 的嵌套层级：`data.outputs.result`。Spring Boot 的 `DifyServiceImpl.parseSseStream()` 正是从 `data.outputs.result` 路径提取最终结果（代码：`root.path("data").path("outputs").path("result").asText()`）。

def import_jobs(replace=False):

### 13.11 关键设计决策汇总

| 决策 | 选择 | 替代方案 | 理由 |
|------|------|----------|------|
| LLM 调用方式 | LangChain ChatOpenAI | 直接 HTTP 调用 | 封装了重试、流式、Prompt 模板，减少样板代码 |
| 嵌入模型 | 远程 text-embedding-v3 | 本地 all-MiniLM | 避免 80MB 模型下载，中文语义效果更好 |
| 向量存储 | ChromaDB 持久化 | FAISS / Milvus | 轻量、零配置、Python 原生，适合小规模部署 |
| API 格式 | 完全兼容 Dify SSE | 自定义 REST API | 原 IRAS 系统零代码改动 |
| 数据库策略 | 新库 iras_v2 + 原表结构 | 修改原 iras 库 | 可独立演化，不影响毕设答辩环境 |
| 去重键 | (job_name, company_name) | 全文哈希 | 同岗位多次出现视为更新而非重复插入 |
| 向量同步 | 增量同步（时间戳） | 全量重建 | 大幅减少嵌入 API 调用量，3000+ 岗位只需增量处理 |
| 批次大小 | BATCH_SIZE=10 | 全量一批 | DashScope 嵌入 API 硬限制每次最多 10 条 |
| 孤儿清理 | 差集对比 | 仅依赖软删除 | 处理 Spring Boot 硬删除（DELETE）产生的孤儿向量 |
| 结果包装 | `{"result": "<think>...</think>[...]"}` | 纯 JSON 数组 | 推理过程可审计，前端可独立展示 think 内容 |
| 流式输出 | 收集全部→分发 text_chunk + final | 实时分组 | 最后一片需作为完整结果放入 workflow_finished |
| 时间戳 | 应用层+数据库层双重默认值 | 仅应用层 | 防止 Spring Boot 漏写导致 NULL 值（NULL 排序在 DESC 最末） |
| 数据来源 | CSV 导入 + 管理后台手动 | 自动采集 | 国内招聘网站均有严格反爬机制，自动化不可行 |
| 简历检索 | LLM 提取关键词 → ChromaDB 检索 | 简历全文直接检索 | 全文含噪音信息（人名/校名/项目），关键词查准率更高 |
